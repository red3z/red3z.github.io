B+树

存储引擎是啥东西?

存储引擎决定了数据在磁盘上的不同组织形式

InnoDB引擎, 支持ACID事务, 不支持外键, 支持表锁和行锁, 索引和数据一起存储, 索引的数据结构都是B+

Myisam引擎, 不支持事务, 不支持外键, 只支持表锁, 索引和数据分开存储

Memory引擎, 数据存在内存中, 处理速度快, 安全性不高

Mysql索引为什么不用Hash表?

  1. hash算法不合适引起数据散列不均匀, 退化成链表
  2. hash表范围查询效率低
  3. 需要大量内存空间

为什么不用二叉树AVL树红黑树?

二叉树分支只有两个, 树太高, IO次数多, 则效率低,

B+树比较矮, 非叶子节点只存放键，不存放值, 一次读取，可以在内存页中获取更多的键，有利于更快地缩小查找范围。

B+树比B树好在哪?

数据只存放在叶子结点, 不存放在非叶子结点, 可以在内存页中获取更多的键，有利于更快地缩小查找范围

叶子与叶子用了 双向链表, 支持了范围查找

查找比较稳定, 便于优化

B+树一般有几层?

千万级数据量3-4层

索引

索引用来做什么?

​ 索引可以让我们查询变得更快

​ 因为更新语句先将数据查询出来修改后再写回磁盘, 因此更新的速度也会更快

索引的分类

  • 聚簇索引(一级索引), 非聚簇索引( 二级索引): 主键索引也被称为聚簇索引, 聚簇索引的叶子结点是整行数据, 非聚簇索引的叶子结点内容是主键的值. ==聚簇指的是和数据聚簇==
  • 单列索引, 联合索引(复合索引)

索引的类型

  • 主键索引, 唯一+非空
  • 唯一索引, 唯一约束, 不允许重复, 一个表允许多个列创建唯一索引
  • 普通索引: 可以重复, 允许为null

表中没有主键的情况下, InnoDB会怎么做?

InnoDB数据和索引放在一起(同一个文件), 而且是必须绑定

  • 若没有主键, 用第一个唯一索引作为聚簇索引
  • 既没有主键, 也没有唯一键, 用6字节的rowID作为聚簇索引

回表是什么?

从某个索引的叶子结点获取聚簇索引的id值, 根据id再去获取数据, 非聚簇索引才需要回表;

尽量减少回表

每个字段都建立索引行不行?

索引占空间

离散度低的数据字段不适合做索引, 比如性别. 建议直接分表存储(建个男表, 建个女表)

创建索引的原则

==最左前缀匹配原则==，组合索引A+B, B是基于A才会有序, 所以单独查A的话就不走索引.

mysql会一直从左向右匹配直到遇到范围查询(>、<、between、like)就停止匹配，比如a = 1 and b = 2 and c > 3 and d = 4 如果建立(a,b,c,d)顺序的索引，d是用不到索引的，如果建立(a,b,d,c)的索引则都可以用到d，a,b,d的顺序可以任意调整。

尽量的扩展索引，不要新建索引。

外键的数据列一定要建立索引

id索引要不要自增?主键使用自增ID还是UUID?

要, 可以减少页分裂, 页分裂操作比较耗时

索引覆盖是什么?

==InnoDB非主键索引的叶子存储的是主键和其他带索引列数据==

(name建立了索引)select id, name from table where name='red';

从索引的叶子节点能获取到全量查询列, 就是索引覆盖, 少了回表操作, 因此速度更快

索引下推是什么?

![image-20240803232143100](/Users/zhangzizhao/Library/Application Support/typora-user-images/image-20240803232143100.png)

(name, age是组合索引)select * from table where name=red and age=10;

没有索引下推: 根据name在存储引擎拿到全量数据, 将数据读取到server层, 按照age做数据过滤

有索引下推: 根据name, age两个列在存储引擎筛选数据

从server层做的数据过滤层下推到存储引擎层

事务

事务的四大特性是什么? 怎么实现的?

  • Atomic原子性: ==UndoLog==实现

    • 如果出现问题, 就回滚到执行之前的状态, 执行之前的状态用UndoLog记录下来

  • Isolation隔离性: ==MVCC实现==, 并发访问数据库时，一个用户的事务不被其他事务所干扰，各并发事务之间数据库是独立的；

  • Durability持久性: ==redolog实现==

    • MySQL写数据先往Buffer中写, (期间可能MySQL宕机, 数据丢失)再同步到磁盘上. RedoLog记录写数据的记录
    • 为什么RedoLog比Buffer快? 1.Buffer数据持久化是随机IO, 每次修改的位置是随机的, RedoLog是文件尾部追加模式, 顺序IO; Buffer以16KB页为单位写入, RedoLog只需要写需要的部分, 无效IO减少

  • Consistency一致性: 完整性和准确性

脏读, 幻读, 不可重复读

  • 脏读: 一个事务读取另一个事务修改了的数据, ‌如果修改事务回滚, ‌那么读取到的数据就是“脏”的, ‌它反映从未真正发生过的数据状态。‌
  • 幻读: 当一个事务读取某些行后，‌由于另一个事务插入新行，‌导致事务A在后续读取时“看到”了这些“幻影”行。‌这通常发生在一个事务需要多次执行相同的查询以获取新的数据行，‌但由于其他事务的插入操作，‌这些新插入的行在事务A的后续查询中被“看到”，‌尽管它们在事务A首次查询时并不存在
  • 不可重复读: 在一个事务内，‌多次读取同一数据时，‌由于其他事务的修改提交，‌导致后续读取的数据和先前读取的数据不一致, ‌这通常发生在事务A读取了某些数据，‌事务B同时更新了这些数据并提交，‌导致事务A再次读取时数据已经改变。‌

事务的隔离级别

  1. Read uncommitted读未提交
  2. Read committed读已提交: 事务只能看到其他事务 ==已经提交的数据更改==。
  3. Repeatable read可重复读(MySQL默认, select id >2, 两次查询不同, 开启了间隙锁, 将id>2的数据锁住): 在同一事务内的多次相同查询始终返回相同的结果, 即使其他事务期间对数据进行了修改并提交; 用Next-Key Lock实现
  4. Serializable串行化

MVCC

Muiti-Version Concurrency Control, 多版本并发控制.

MVCC用来干啥?

MVCC主要处理读请求, 指的是快照读, 乐观的. (与之相反的是当前读, 需要加锁)
